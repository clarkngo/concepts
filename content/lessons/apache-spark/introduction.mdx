## Intuition

When data **does not fit on one machine**, you still want code that reads like everyday analytics: filter rows, join tables, roll up counts. **Apache Spark** is a **cluster computing framework** that lets you express that work with high-level APIs (SQL, DataFrames, Datasets) and then **parallelizes** it across **executors** on many nodes.

Think of Spark as **not** “a bigger database.” It is an **execution engine**: it **builds a plan**, **schedules** work as tasks, and **recomputes** or retries when things fail—while usually **reading and writing** data in external systems (object storage, data lakes, warehouses) rather than owning storage.

Three ideas to keep in front of mind:

- **Express once** — you describe *what* you want (transformations).
- **Run distributed** — the runtime figures out *how* to spread work.
- **Recover** — lineage and task retries are first-class, not an afterthought.

## The core loop: driver, executors, and SparkSession

- **Driver** — runs your program, hosts the **SparkContext** / session, builds the plan, and coordinates stages; it is **not** where bulk data lives.
- **Executors** — run **tasks** on worker nodes (CPU, memory, local disk for spills); they hold **cached** partitions when you `cache()` / `persist()`.
- **SparkSession** (`spark`) — the single entry point in modern Spark (Spark 2+), replacing the older split between `SQLContext` and `HiveContext`.

## Lazy evaluation (transformations vs actions)

Most Spark **transformations** are **lazy**: `filter`, `select`, `join`, and friends update a **logical plan**—a recipe—not a result. **Nothing large runs** until you call an **action** (`count`, `show`, `collect`, `write`, …), which **triggers** execution.

That delay is deliberate. Spark can **optimize** before reading data: combine filters, drop unused columns (**column pruning**), push predicates toward storage, and apply **whole-stage code generation** so hot paths run as tight loops instead of row-by-row interpreter overhead.

**Rule of thumb:** transformations **describe**; actions **commit** (and clock time usually starts mattering in earnest).

## What to remember

- Spark is an **engine** for **distributed execution**, usually layered on **external storage**.
- **Transformations** build a plan; **actions** run it.
- Laziness exists so Spark can **optimize** before touching data at scale.
