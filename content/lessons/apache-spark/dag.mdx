## Intuition

A single-threaded script runs **roughly** top to bottom. Spark instead compiles a **job** into a **graph of dependencies**: which outputs must exist before which tasks run. That graph is a **DAG**—**directed** edges, **no cycles**—so the scheduler can pick a **consistent order**, run independent branches **concurrently**, and **retry** failed work without contradicting “must happen first.”

People say “the DAG” in two different voices; keep them separate:

1. **Lineage / logical plan** — what your transformations **mean**, and how a dataset was **derived** (useful for optimization and **recomputation** after loss).
2. **Execution DAG** — how Spark **materializes** that intent into **stages** and **tasks**, usually **cut** at **shuffle** boundaries where partitions are **redistributed**.

<KeyIdea>
  A **DAG** encodes **dependencies**. Spark uses it to decide **parallelism** (what can overlap) and **barriers** (what must wait)—especially where a **shuffle** forces all-to-all data movement between stages.
</KeyIdea>

## DAG in graph theory (the relationship)

A **directed acyclic graph (DAG)** has:

- **Directed** edges — “A must finish before B.”
- **No directed cycles** — you cannot follow edges forward from a node and return to it.

Spark’s **stage/task** dependency graph is built acyclic so dependencies induce a **partial order**: if task T2 needs partitions from T1, T1’s shuffle write must complete before T2’s shuffle read proceeds.

## Stages, tasks, and shuffle boundaries

Spark groups work into **stages** separated by **shuffle** exchanges (for example `groupBy`, aggregations that repartition by key, many **joins** when both sides are large). **Inside** a stage, **narrow** dependencies often **pipeline**: several row-wise operations can run in one task without a cluster-wide shuffle.

Roughly:

- **Narrow** (map-like) — each output partition depends on **at most one** input partition; work can **fuse** in one task.
- **Wide** (shuffle) — partitions depend on **many** upstream partitions; Spark must **write** shuffle blocks and **read** them in the next stage.

So the **execution DAG** is **not** “each `map` is a node.” It is **stages and tasks** after planning—coarser than the logical plan, finer than “one blob per job.”

- **Lineage** (RDD lineage, logical plan graph) — **derivation** for **fault tolerance** and **Catalyst** optimizations.
- **Stage DAG** — **runtime scheduling**: which **TaskSets** are eligible, and where **shuffle barriers** sit.

Same **graph** metaphor, different **question**: *recompute and optimize* versus *execute under cluster constraints*.

## Why “acyclic” matters in practice

Cycles in the **scheduling** graph would make “next task” **ill-defined**. Your **driver** can still **loop** over Spark jobs or batches; each **job’s** internal task graph stays **acyclic** so progress and retries stay well-defined.

## What to remember

- **DAG** = directed dependencies, **no cycles** — enables **order** and **safe retries**.
- **Shuffles** usually **split stages**; **narrow** work **pipelines** inside a stage.
- **Lineage** and **stage DAG** are **related graphs**, not the **same** picture.
