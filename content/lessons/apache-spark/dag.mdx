## Intuition

When Spark runs a job, it does not “just run lines top to bottom” like a single-threaded script. It turns your job into a **graph of work** with dependencies. That graph is constrained to be **acyclic**—no cycles—so the scheduler can **order** work safely and **retry** failed tasks without contradicting dependencies.

This lesson separates two related ideas people casually call “the DAG”:

1. **The lineage / logical plan** (what your transformations *mean*).
2. **The execution DAG** (how Spark *cuts* that plan into **stages** and **tasks** at shuffle boundaries).

<KeyIdea>
  A **DAG** is the shape of *dependencies*. Spark uses that shape to decide **what can run in parallel** and **what must wait**—especially at **shuffle** boundaries where partitions are exchanged across the cluster.
</KeyIdea>

## DAG in graph theory (the relationship)

A **directed acyclic graph (DAG)** is:

- **Directed**: edges have a direction (A must happen before B).
- **Acyclic**: you cannot follow edges and return to the same node—**no cycles**.

Spark’s runtime scheduling graph is built to be acyclic so dependencies define a **partial order**: if task T2 depends on output partitions produced by T1, then T1 must finish first.

## Where Spark builds a DAG: stages and shuffle boundaries

Spark groups work into **stages** separated by **shuffle** operations (for example `groupByKey`, repartitioning joins, wide dependencies). Inside a stage, **pipelined** narrow transformations can often run without an all-to-all shuffle.

Roughly:

- **Narrow** transformations (map-like): can be **pipelined** in the same stage when dependencies are narrow.
- **Wide** transformations (shuffle-like): typically **end** a stage and **start** a new one because partitions must be exchanged across the cluster.

So the **execution DAG** is not “every `map` is a node.” It is closer to: **a graph of stages/tasks** produced after planning and partitioning decisions.

- **Lineage (RDD lineage / logical plan lineage)** helps Spark recompute lost partitions after failures by tracking **how** a dataset was derived.
- **The scheduler’s DAG of stages** is what the **DAGScheduler** uses to submit **TaskSets** to the **TaskScheduler**.

Both are graph-shaped stories, but they answer different questions: *derivation for recomputation* vs *parallel execution with shuffle cuts*.

## Why “acyclic” matters in practice

If cycles existed in the scheduling graph, you could not define a consistent execution order for tasks. Spark’s programming model still lets you *iterate* algorithms (loops in your driver program), but the **per-job** execution graph that the scheduler reasons about remains acyclic at the stage/task dependency level.

## What to remember

- **DAG** = directed dependencies + **no cycles** (mathematical DAG).
- Spark uses DAG-shaped **execution plans**; **shuffles** split **stages**.
- **Lineage** and **stage DAG** are related but **not the same diagram**.
